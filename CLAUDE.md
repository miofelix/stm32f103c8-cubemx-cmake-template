# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Minimal macOS-only CMake firmware template for the **STM32F103C8Tx** (Cortex-M3, 64KB flash / 20KB RAM, LQFP48), generated by STM32CubeMX 6.17. The CubeMX project name is deliberately kept generic (`stm32f103_project`) so the template can be copied without inheriting a feature-specific name in the `.ioc` and build artifacts.

This repository is intentionally maintained for **macOS + Homebrew only**. Do not add Linux/Windows-specific setup, CI, or documentation unless the project scope changes.

System clock runs at **72 MHz**, derived from an 8 MHz external crystal (HSE → PLL ×9). See `SystemClock_Config()` in `Core/Src/main.c`.

## Build & flash

```sh
cmake --preset Debug             # configure (also: Release)
cmake --build --preset Debug     # build -> build/Debug/stm32f103_project.{elf,hex,bin,map}
cmake --workflow --preset Debug  # configure + build in one step

cmake --build --preset Debug --target flash   # flash via OpenOCD (ST-LINK default)
cmake --build --preset Debug --target erase   # mass erase
cmake --build --preset Debug --target reset   # reset target
```

- Toolchain: `arm-none-eabi-gcc`, generator Ninja. Requires macOS + Homebrew packages `cmake`, `ninja`, `open-ocd`, and cask `gcc-arm-embedded` on PATH. CMake must be **3.25+** because `CMakePresets.json` uses preset schema version 6 and workflow presets.
- Flash adapter override (per-command): `FLASH_ADAPTER=daplink cmake --build --preset Debug --target flash` (values: `stlink`, `daplink`).
- OpenOCD target override: `OPENOCD_TARGET=target/stm32f1x.cfg ...`.
- `.hex`/`.bin` are produced as a POST_BUILD step on every link (see `cmake/firmware-images.cmake`).
- There are no automated unit tests; verification is done by CMake builds and on-target flash. CI (`.github/workflows/build.yml`) runs on macOS, builds both presets on push/PR, uploads firmware images as artifacts, and smoke-tests `scripts/rename.sh` by building a renamed temporary copy.

## Tooling & DX

- **Debug builds enable `USE_FULL_ASSERT`** (HAL/LL `assert_param` checking), wired through `target_compile_definitions(stm32cubemx INTERFACE ...)` in the root `CMakeLists.txt` so it reaches application *and* driver sources. A failed check halts in `assert_failed()` (`Core/Src/main.c`, `USER CODE BEGIN 6`). `Core/Inc/stm32_assert.h` routes the LL drivers' asserts to the same handler. Release builds define nothing, so asserts compile out. Because the define lives in CMake (not `hal_conf.h`), it survives CubeMX regeneration.
- **`.vscode/`** ships shared `launch.json` (Cortex-Debug + OpenOCD one-click debug), `tasks.json` (build/flash/erase/reset), and `extensions.json`. These are whitelisted in `.gitignore`.
- **`.clangd`** points clangd at `build/Debug/compile_commands.json` so vendor headers resolve; run `cmake --preset Debug` once to generate the database. IncludeCleaner hints are disabled (too noisy on HAL).
- **`scripts/rename.sh <new_name>`** renames the project (CMake target, the `.ioc` fields and filename, `.vscode/launch.json`, and the docs). It reads the current name from `CMakeLists.txt`, so it is re-runnable; it does **not** retarget the MCU. CI validates this workflow.
- **`scripts/check-template.sh`** verifies the project-name/`.ioc` pairing, required CubeMX generation settings, ignored `.mxproject` metadata, and that the root CMake file does not take over HAL source selection. Run it after every CubeMX regeneration.

## Architecture & where code goes

The build is a two-layer CMake setup:

- **`CMakeLists.txt` (root)** — the editable top-level. Add **your own** sources, includes, defines, and libraries here, in the clearly marked `# Add user ...` slots. CubeMX creates it once; preserve the template-owned helper includes and Debug assert definition when regenerating.
- **`cmake/stm32cubemx/CMakeLists.txt`** — CubeMX-generated. Lists all generated application sources (`MX_Application_Src`), HAL/CMSIS driver sources (`STM32_Drivers`), include dirs, and symbols (`USE_HAL_DRIVER`, `STM32F103xB`). **Regenerated by CubeMX** — do not hand-edit; let CubeMX manage it.
- `cmake/gcc-arm-none-eabi.cmake` — active toolchain (compiler/linker flags, `-mcpu=cortex-m3`, `nano.specs`, links the linker script). `cmake/starm-clang.cmake` is an alternative ST clang toolchain; switch by changing `toolchainFile` in `CMakePresets.json`.
- `STM32F103XX_FLASH.ld` — linker script (memory regions, sections). `startup_stm32f103xb.s` — vector table and reset handler.

Source layout:
- `Core/Inc`, `Core/Src` — application code (CubeMX-managed peripheral init + `main.c`).
- `Drivers/STM32F1xx_HAL_Driver`, `Drivers/CMSIS` — vendor HAL and CMSIS. Treat as read-only.

Ownership boundary:

- **CubeMX-owned firmware code:** `Core/`, `Drivers/`, `cmake/stm32cubemx/`, `startup_*.s`, `*.ld`, and generated sections of the root `CMakeLists.txt`.
- **Tracked source configuration:** `<project>.ioc`. It is the source of truth for clocks, pins, peripherals, middleware, HAL modules, and generated source selection.
- **Template-owned assets:** root CMake extensions, `CMakePresets.json`, helper modules under `cmake/`, `scripts/`, `.github/`, `.vscode/`, and documentation.
- **Local CubeMX metadata:** `.mxproject` is ignored and must not be committed.

## HAL and CubeMX contract (important)

The HAL package in `Drivers/`, module switches in
`Core/Inc/stm32f1xx_hal_conf.h`, generated initialization/MSP code, and the
`STM32_Drivers` source list form one CubeMX-managed unit. The repository does
not vendor or compile the complete STM32F1 HAL independently.

To add or remove a peripheral, update `<project>.ioc` in STM32CubeMX and
regenerate. Do not hand-enable `HAL_<P>_MODULE_ENABLED`, copy individual vendor
drivers, glob `Drivers/.../Src`, or mutate the generated `STM32_Drivers` target
from the root CMake file. This keeps headers, sources, module defines, pin mux,
clocks, DMA, NVIC, and MSP initialization consistent.

## CubeMX code-generation contract (important)

The `.ioc` must retain `KeepUserCode=true`, `DeletePrevious=true`,
`LibraryCopy=1`, and `TargetToolchain=CMake`. `LibraryCopy=1` deliberately keeps
only the HAL/CMSIS files required by the current configuration; large driver
deletions after regeneration are expected when peripherals are removed.

`Core/` files are owned by CubeMX. When the `.ioc` is regenerated, **everything
outside the `USER CODE BEGIN X` / `USER CODE END X` markers is overwritten.**
Only write application code inside those marker blocks. Editing peripheral
configuration (clocks, pins, DMA, NVIC) should be done in CubeMX via the `.ioc`,
not by hand-editing generated initialization code.

When adding application modules, prefer creating new files and registering them in the **root** `CMakeLists.txt` (not the CubeMX-generated one), so they survive regeneration.

After regeneration, run:

```sh
scripts/check-template.sh
cmake --workflow --preset Debug
cmake --workflow --preset Release
```

Review generated changes as a coherent set. Do not restore deleted HAL files
unless the corresponding peripheral is still enabled in the `.ioc`. If
retargeting to another STM32, regenerate the startup file, linker script, MCU
symbol, CPU flags, HAL/CMSIS package, and generated CMake target together.
