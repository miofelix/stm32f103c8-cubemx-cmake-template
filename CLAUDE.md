# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Minimal CMake firmware template for the **STM32F103C8Tx** (Cortex-M3, 64KB flash / 20KB RAM, LQFP48), generated by STM32CubeMX 6.17. The CubeMX project name is deliberately kept generic (`stm32f103_project`) so the template can be copied without inheriting a feature-specific name in the `.ioc` and build artifacts.

System clock runs at **72 MHz**, derived from an 8 MHz external crystal (HSE → PLL ×9). See `SystemClock_Config()` in `Core/Src/main.c`.

## Build & flash

```sh
cmake --preset Debug          # configure (also: Release)
cmake --build --preset Debug  # build -> build/Debug/stm32f103_project.{elf,hex,bin,map}

cmake --build --preset Debug --target flash   # flash via OpenOCD (ST-LINK default)
cmake --build --preset Debug --target erase   # mass erase
cmake --build --preset Debug --target reset   # reset target
```

- Toolchain: `arm-none-eabi-gcc`, generator Ninja. Requires `cmake ninja openocd arm-none-eabi-gcc` on PATH.
- Flash adapter override (per-command): `FLASH_ADAPTER=daplink cmake --build --preset Debug --target flash` (values: `stlink`, `daplink`).
- OpenOCD target override: `OPENOCD_TARGET=target/stm32f1x.cfg ...`.
- `.hex`/`.bin` are produced as a POST_BUILD step on every link (see `cmake/firmware-images.cmake`).
- There are no automated tests; verification is done on-target via flash.

## Architecture & where code goes

The build is a two-layer CMake setup:

- **`CMakeLists.txt` (root)** — the editable top-level. Add **your own** sources, includes, defines, and libraries here, in the clearly marked `# Add user ...` slots. This file is generated once and never regenerated by CubeMX.
- **`cmake/stm32cubemx/CMakeLists.txt`** — CubeMX-generated. Lists all generated application sources (`MX_Application_Src`), HAL/CMSIS driver sources (`STM32_Drivers`), include dirs, and symbols (`USE_HAL_DRIVER`, `STM32F103xB`). **Regenerated by CubeMX** — do not hand-edit; let CubeMX manage it.
- `cmake/gcc-arm-none-eabi.cmake` — active toolchain (compiler/linker flags, `-mcpu=cortex-m3`, `nano.specs`, links the linker script). `cmake/starm-clang.cmake` is an alternative ST clang toolchain; switch by changing `toolchainFile` in `CMakePresets.json`.
- `STM32F103XX_FLASH.ld` — linker script (memory regions, sections). `startup_stm32f103xb.s` — vector table and reset handler.

Source layout:
- `Core/Inc`, `Core/Src` — application code (CubeMX-managed peripheral init + `main.c`).
- `Drivers/STM32F1xx_HAL_Driver`, `Drivers/CMSIS` — vendor HAL and CMSIS. Treat as read-only.

## CubeMX code-generation contract (important)

`Core/` files are owned by CubeMX (`KeepUserCode=true`). When the `.ioc` is regenerated, **everything outside the `USER CODE BEGIN X` / `USER CODE END X` markers is overwritten.** Only write application code inside those marker blocks. Editing peripheral configuration (clocks, pins, NVIC) should be done in CubeMX via `stm32f103_project.ioc`, not by hand-editing the generated init code.

When adding application modules, prefer creating new files and registering them in the **root** `CMakeLists.txt` (not the CubeMX-generated one), so they survive regeneration.
